# Настройка Supabase — шаги перед первым запуском

## 1. Выполнить SQL-схему

1. Откройте проект на supabase.com → **SQL Editor** → **New query**
2. Скопируйте содержимое файла `supabase_schema.sql` целиком
3. Нажмите **Run**

Это создаст три таблицы (`profiles`, `orders`, `returns`), все политики безопасности (RLS) и включит Realtime.

## 2. Отключить подтверждение email (важно для внутреннего инструмента)

По умолчанию Supabase требует подтверждение email при регистрации — для внутреннего цехового инструмента это не нужно.

1. **Authentication** → **Providers** → **Email**
2. Выключите **"Confirm email"**
3. Сохраните

Без этого шага новые сотрудники не смогут войти сразу после создания — потребуется подтверждение несуществующего email.

## 3. Создать первого администратора (Сергея)

Поскольку создание сотрудников через интерфейс дашборда теперь требует, чтобы хотя бы один администратор уже существовал, первого нужно создать вручную:

1. Откройте дашборд (`index.html`) в браузере
2. На экране входа браузер покажет ошибку — это нормально, профилей ещё нет
3. Откройте консоль разработчика (F12) и выполните:

```js
await sb.auth.signUp({
  email: 'sergey@internal.lab',
  password: 'ВАШ_НАДЁЖНЫЙ_ПАРОЛЬ',
  options: { data: { name: 'Сергей', role: 'Администратор', is_admin: true } }
});
```

4. После этого войдите в дашборд: логин `Sergey`, пароль — тот, что указали выше

## 4. Дальше — всё через интерфейс

Войдя как администратор, Сергей создаёт остальных сотрудников через вкладку **«👥 Сотрудники» → «+ Добавить сотрудника»**. Система сама сгенерирует временный пароль — его нужно будет сообщить сотруднику лично (показывается один раз в уведомлении после создания).

## Важно про безопасность

- Паролей в коде дашборда теперь нет вообще — все хранятся хэшированными в Supabase Auth
- RLS-политики гарантируют: подзаказ может "взять" только если он реально свободен — гонки между сотрудниками больше невозможны на уровне базы
- Realtime-подписка заменяет периодический опрос — обновления приходят мгновенно
